# 工具设计哲学

## 核心原则：工具是 Agent 的双手

工具的描述（description）与其实现同等重要。LLM 通过阅读描述来决定 *何时* 使用工具，因此模糊的描述会导致工具选择错误。

## 描述最佳实践

1. **第一句话明确用途**
2. **添加 "use when" 指引**："当需要发现有哪些可用文件时使用此工具"
3. **添加 "prefer X when" 对比**："当需要在多文件中搜索特定内容时，优先使用 search_files"
4. **文档化参数并附上示例**

## 错误设计

每个工具错误包含：
- `error_code`：机器可读的枚举值（如 `not_found`、`path_outside_sandbox`）
- `message`：人类可读的说明
- `hint`：可操作的建议，帮助模型自我纠正

这将错误从"死胡同"变为 Agent 的学习机会。

## 经验教训

- 对比式描述（"当 A 时用 X，当 B 时优先用 Y"）是提升工具选择准确率的最大杠杆
- 带有下一步建议的错误提示（如"先用 list_files 发现可用文件"）为模型提供恢复路径，减少循环
- 带 `dry_run` 参数的工具让 Agent 先规划再执行 —— 对破坏性操作至关重要
- 格式特定的工具（read_pdf、read_docx）优于单一通用 read_file，因为模型能根据文件扩展名匹配工具
- 在 Registry 层设置权限守卫能统一拦截所有工具调用 —— 无需在每个工具内部加检查
